# Contributing

Спасибо за интерес к проекту! Вклад в проект приветствуется. Ниже приведены рекомендации по участию в разработке.

## Как внести вклад

Вы можете помочь проекту несколькими способами:

- **Баг-репорты:** Сообщайте о найденных ошибках
- **Фича-реквесты:** Предлагайте новые функции
- **Исправление багов:** Исправляйте найденные ошибки
- **Улучшение документации:** Дополняйте и исправляйте документацию
- **Написание тестов:** Добавляйте новые тесты
- **Оптимизация кода:** Улучшайте производительность и читаемость кода

## Процесс внесения изменений

1. **Форк проекта**
   - Нажмите кнопку "Fork" в верхнем правом углу репозитория на GitHub
   - Склонируйте свой форк локально:
     ```bash
     git clone https://github.com/ваш-пользователь/My-CV-Engineering-Projects.git
     cd My-CV-Engineering-Projects
     ```

2. **Создание ветки**
   - Создайте новую ветку для вашего изменения:
     ```bash
     git checkout -b feature/ваша-фича
     # или
     git checkout -b bugfix/описание-бага
     ```

3. **Внесение изменений**
   - Внесите необходимые изменения в код
   - Убедитесь, что код соответствует стилю проекта
   - Добавьте комментарии, если необходимо

4. **Тестирование**
   - Запустите тесты, чтобы убедиться, что изменения не сломали существующий функционал:
     ```bash
     pytest tests/
     ```
   - При необходимости добавьте новые тесты для нового функционала

5. **Коммит изменений**
   - Добавьте измененные файлы:
     ```bash
     git add .
     ```
   - Сделайте коммит с описательным сообщением:
     ```bash
     git commit -m "Описание изменений"
     ```

6. **Push изменений**
   - Загрузите изменения в свой форк:
     ```bash
     git push origin feature/ваша-фича
     ```

7. **Создание Pull Request**
   - Перейдите на страницу своего форка на GitHub
   - Нажмите кнопку "Compare & pull request"
   - Заполните шаблон PR, описав изменения
   - Укажите, какие задачи решает ваш PR
   - Назначьте ревьюеров, если необходимо

## Требования к коду

- Следуйте стилю кодирования PEP 8
- Добавляйте типизацию, где это возможно
- Пишите понятные имена переменных и функций
- Добавляйте docstrings для всех публичных функций и классов
- Минимизируйте дублирование кода
- Пишите чистый и поддерживаемый код

## Тестирование

- Все новые функции должны быть покрыты тестами
- Тесты должны находиться в папке `tests/`
- Используйте `pytest` для запуска тестов
- Стремитесь к высокому покрытию кода тестами

## Документация

- Обновляйте документацию при добавлении нового функционала
- Документация должна быть на английском языке
- Используйте Markdown для форматирования
- Обновляйте соответствующие разделы в папке `docs/`

## Рецензирование кода

- Будьте вежливы и уважительны при обсуждении кода
- Предлагайте альтернативные решения, если считаете нужным
- Обосновывайте свои предложения
- Будьте открыты к обратной связи
